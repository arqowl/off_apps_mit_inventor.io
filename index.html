02 — Interface de Usuário

O que é Interface?

Interface é a ponte entre humano e máquina. É tudo que o usuário vê, toca e compreende.

👁
Visual

O que o usuário vê — cores, formas, tipografia

🖐
Interativo

Com o que o usuário interage — botões, gestos, arrastar

🧠
Mental

O que o usuário entende — sem precisar ler manual

"Uma boa interface é invisível — você só a percebe quando ela é ruim."

Princípio de Design de Interação

Exemplo: Imagina um elevador sem nenhuma indicação de onde ficam os botões. Ele funciona — mas é uma experiência terrível.

03 — UI: Convenção vs. Criatividade

Por que o "Genérico" é Poderoso?

✅ Convenção Funciona
Vermelho → Parar / Perigo / Excluir
🔍
Lupa → Buscar (em qualquer app do mundo)
✕
X no canto → Fechar janela/modal
☰
Hambúrguer → Menu de navegação
⚠️ "Inovação" sem contexto

Renomear "Fechar" como "Ejetar da Experiência" parece criativo, mas o usuário trava. O custo cognitivo de aprender uma nova convenção é alto demais.

Inovar na forma de interagir pede que o usuário desaprenda. Isso cria atrito.

Foco: No nosso app, os botões START e PAUSE já comunicam tudo antes do usuário tocar. Essa é a força do genérico.

💡

Princípio: Convenção vence criatividade quando o objetivo é usabilidade. Inove no conteúdo, não na navegação.

06 — Controle de Fluxo

Se... Então... Senão

// Exemplo real do nosso jogo
SE borda = -1 ← borda inferior
  ENTÃO
    Pontos.texto ← "GAME OVER"
    Bola.ativo ← FALSO
    PlaySom("note-do.mp3")
SENÃO
    Bola.Bounce(borda)
    PlaySom("single-knock.mp3")

Atenção: O computador executa UM caminho — o Então ou o Senão. Nunca os dois ao mesmo tempo.

Lógica Booleana — VERDADEIRO ou FALSO

Qualquer condição tem só dois estados. Sem meio-termo.

E
Ambas as condições precisam ser VERDADEIRO
OU
Pelo menos uma precisa ser VERDADEIRO
NÃO
Inverte: VERDADEIRO vira FALSO

No jogo:
Avançado.Marcado E Pontos MOD 2 = 0
→ Ambas verdadeiras = acelera a bola!

Superpower: detectar par/ímpar e criar ciclos automáticos.

Por que o MOD é tão útil?

Detectar Par / Ímpar
n MOD 2 = 0 → PAR ✓
n MOD 2 = 1 → ÍMPAR
No Nosso Jogo
SE Pontos MOD 2 = 0
  E Avançado = VERDADEIRO
  Velocidade ← Velocidade + 1
← a cada 2 pontos, bola acelera!

Analogia: Pense em rodadas de 5. Na 5ª, 10ª, 15ª o contador "zera". O MOD detecta exatamente esses momentos.

08 — Variáveis

A Memória do App

Uma variável é como uma caixinha com etiqueta. O nome é a etiqueta; o valor é o conteúdo. O app lembra e muda esse valor enquanto roda.

Global vs. Local: Global = qualquer parte do app acessa. Local = existe só dentro do procedimento que a criou. Use global para dados que precisam persistir.

09 — Procedimentos

Atalhos com Nome

Um procedimento é um bloco de ações que você nomeia e reutiliza. Cria uma vez, chama onde quiser. A divisão fundamental: com ou sem retorno.

Sem Retorno — só executa ações
procedimento PlaySom(arquivo)
  SE Som.marcado
    Som1.fonte ← arquivo
    Som1.Tocar()
← não devolve nada para quem chamou

Analogia: "Acenda a luz." — Uma ordem que resulta numa ação. Ninguém espera uma resposta de volta.

Exemplos no jogo
PlaySom() FlagDificuldade() AlterarEMostrarPontuacao()
Com Retorno — calcula e devolve um valor
função calcularMedio(a, b)
  devolver (a + b) / 2

← uso em outro lugar:
media ← calcularMedio(8, 10)
← media agora vale 9

Analogia: "Qual é o dobro de 5?" — Uma pergunta que espera uma resposta. O valor devolvido pode ser usado em outro cálculo.

💡 No App Inventor: bloco "to … result" devolve valor · "to … do" só executa

10 — Tudo se Conecta

Ping Pong: A Arquitetura

Evento START.Clicado

Centraliza bola + raquete · direção aleatória 225°–315° · zera placar

↓
Evento Canvas.Arrastado

Raquete.x ← posição X do toque. Só o eixo horizontal se move.

↓
Evento Bola.Colidiu

Rebate (360 − Heading) · +1 ponto · PlaySom · FlagDificuldade(MOD 2)

↓
Evento Bola.BordaAtingida

Borda -1 (baixo) → GAME OVER. Demais bordas → Bounce.

Procedimentos Criados

Sem retorno AlterarEMostrarPontuacao(NovoPonto)
Sem retorno PlaySom(Arquivo)
Sem retorno FlagDificuldade()
